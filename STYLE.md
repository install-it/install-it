# STYLE.md - Coding Style Guide

Canonical conventions for install-it. Applies to all code in the repository unless a language-specific section says otherwise.

## How To Use This Guide

- Treat this document as team guidance for consistency.
- Apply recommendations with judgment based on context.
- If two rules conflict, apply the priority order below.

## Rule Priority Order

1. Security and access control
2. Correctness and data integrity
3. Existing architecture and file placement
4. Simplicity and readability
5. Local consistency with nearby code

## General Principles

- Prefer simple, direct solutions over premature abstraction.
- **YAGNI Threshold**: Keep logic inline until it exceeds a readable screen (> 100 lines) or needs reuse across entry points.
- Keep logic concise; avoid single-use helpers unless reuse is clear.
- Avoid intermediate variables when direct expressions remain readable.
- Ask for clarification when requirements are ambiguous.

## Naming Conventions

| Context | Convention | Example |
|---|---|---|
| TypeScript functions/methods | `camelCase` | `handleSubmit` |
| Classes/types (TS) | `PascalCase` | `DriverGroup` |
| Constants (TS/Go) | `UPPER_SNAKE_CASE` | `MAX_RETRIES` |
| CSS classes | `kebab-case` | `gap-x-2` |
| Database columns / JSON fields | `snake_case` | `driver_group_id` |
| i18n translation keys | flat `camelCase` with semantic prefix | `toastSaved`, `hwCpu`, `opContain` |

Go uses casing for visibility: `camelCase` is unexported, `PascalCase` is exported.

## Constraints

- Do not modify `.github/workflows/**` without explicit human instruction.
- Do not modify `vendor/**`, `node_modules/**`, or installed dependency contents.
- Do not hand-edit `frontend/wailsjs/**` — it is auto-generated by Wails.
- Do not commit secrets or local `.env` values.
- **Fix with linter/formatter first**: Run `npm run lint` and `npm run format` before manually fixing style issues. Same for Go — run `go fmt` and `go vet` before touching code.

## Frontend: Vue / TypeScript

### Basics

- Use TypeScript, `<script setup lang="ts">`. No Options API.
- Type-based props/emits: `defineProps<{ ... }>()` and `defineEmits<{ ... }>()`
- Block order: `<script>` → `<template>` → `<style>`
- Blank lines between all template tags (`vue/padding-line-between-tags`)

### Inline Typing

Define types inline directly where they are used unless the type is shared across multiple files — then place it in `frontend/src/types/`.

```typescript
// Good — inline for single-use
defineProps<{ title: string }>()

// Good — shared type in types/
import type { Command } from '@/types/execute'
```

### State Grouping

Prefer `ref` over `reactive`. Group related parameters into unified object refs by context:

```typescript
// Good — grouped by concern
const form = ref({ name: '', email: '' })
const ui = ref({ loading: false, showModal: false })

// Avoid — separate refs for related state
const loading = ref(false)
const showModal = ref(false)
```

### Functions

- **Named functions** for component-level methods and root handlers.
- **Arrow functions** strictly for inline handlers, callbacks, and promise chains.

```typescript
// Good — named function for component logic
function handleSubmit() {
  ruleSetStorage.Add(ruleSet.value)
    .then(handleSuccess)
    .catch(reason => toast.add({ title: reason.toString(), color: 'error' }))
}

// Good — arrow function for inline handler
<button @click="() => ui.showModal = true">Open</button>
```

### Async: `.then().catch()` over `async/await`

Prefer promise chains for Wails binding calls. Reserve `async/await` for cases where it genuinely simplifies logic.

```typescript
// Preferred — promise chain
matcher.MatchedGroupIds()
  .then(ids => { /* ... */ })
  .catch(() => toast.add({ title: t('toast.noHardwareInfo'), color: 'error' }))

// Acceptable when it simplifies logic
async function loadDrivers() {
  const groups = await driverGroupStorage.All()
  // ...
}
```

### Layout

Flex or grid layout combined with Tailwind gap utilities (`gap-x-2`, `gap-y-3`). Use Nuxt UI v4 components (`UButton`, `UInput`, `USelect`, `UModal`, etc.) as building blocks.

```vue
<template>
  <form class="flex h-full flex-col gap-y-3">
    <div class="grid grid-cols-12 gap-1">
```

### Wails Bindings

Import from `@/wailsjs/go/<package>/<Struct>`. All bindings return promises.

```typescript
import { TriggerNativeUpdate } from '@/wailsjs/go/update/Updater'
import * as ruleSetStorage from '@/wailsjs/go/storage/RuleSetStorage'

ruleSetStorage.Add(ruleSet.value)
  .then(handleSuccess)
  .catch(reason => toast.add({ title: reason.toString(), color: 'error' }))
```

### i18n

Flat camelCase keys. No nesting, no dots. Prefix describes **what the string is**, not where it appears.

| Prefix | Nature | Example |
|--------|--------|---------|
| *(none)* | Universal UI atoms | `save`, `cancel`, `name`, `path` |
| `action` | User-triggerable actions | `actionShutdown`, `actionReboot` |
| `buildType` | Build/variant identifiers | `buildTypeDev`, `buildTypeProduction` |
| `category` | Classification labels | `categoryDisplay`, `categoryNetwork` |
| `desc` | Help text, hints, tooltips | `descMutuallyExclusive` |
| `err` | Error/failure messages | `errStartFailed`, `errNothingToImport` |
| `field` | Form/table column labels | `fieldAllowedExitCode`, `fieldSource` |
| `hw` | Hardware component names | `hwCpu`, `hwGpu`, `hwNic` |
| `label` | Generic display labels | `labelPleaseSelect`, `labelNA` |
| `msg` | Informational messages | `msgUnsavedChanges`, `msgExitCode` |
| `op` | Match rule operators | `opContain`, `opNotEqual`, `opRegex` |
| `setting` | Setting/config labels | `settingLanguage`, `settingAutoCheckUpdate` |
| `status` / `statusShort` | Execution status (full / abbreviated) | `statusPending`, `statusShortPending` |
| `step` | Workflow/progress step labels | `stepBackup`, `stepExtract` |
| `title` | Page/section/modal titles | `titleSettings`, `titleExecutionStatus` |
| `toast` | Toast notification messages | `toastSaved`, `toastNoUpdate` |
| `tool` | System utility/tool names | `toolDeviceManager`, `toolBluetooth` |
| `warn` | Warning messages | `warnDbNoDrivers` |

Dynamic keys (status values, operator enums, hardware parts) use a helper function — never backtick template literals in Vue template `{{ }}`:

```typescript
function hwKey(part: string): string {
  return `hw${part.charAt(0).toUpperCase() + part.slice(1)}`
}
```

### Pinia Stores

Use setup stores (factory function returning refs). No Options API.

```typescript
// Good — setup store
export default defineStore('driverGroup', () => {
  const groups = ref<storage.DriverGroup[]>([])
  return { groups }
})
```

## Backend: Go

### General

- Use `go fmt` before committing. Run `go vet` on changed packages.
- Keep execution logic straightforward and procedural until reuse is clear (same YAGNI principle as frontend).
- Return errors explicitly; do not silently ignore with `_`.

### Error Handling

```go
// Good — explicit error check
if err := db.Open(); err != nil {
    return fmt.Errorf("opening db: %w", err)
}

// Avoid — ignoring errors
_ = db.Open()
```

### Database

- Use `glebarez/sqlite` (CGO-free). Do not introduce CGO-dependent drivers.
- Schema changes go through `gormigrate` migrations in `pkg/storage/db.go`. Never call `AutoMigrate` directly outside a migration.
- `Porter` closes and reopens the DB on import — if adding storage code, account for this lifecycle.

### Wails Bindings

New Go structs exposed to the frontend must be added to `Bind` in `main.go`. Enums go in `EnumBind`.

```go
// In main.go
Bind: []interface{}{
    app,
    &storage.DriverGroupStorage{...},
    // new struct here
},
EnumBind: []interface{}{
    // enum values here
},
```

## Quick Review Checklist

**All code:**
- Logic is inline unless it exceeds ~100 lines or needs reuse
- No intermediate variables when direct expressions suffice
- Errors are handled explicitly
- `go fmt` / `go vet` applied (Go); `npm run lint` / `npm run format` pass (Vue/TS)

**Frontend (Vue/TS):**
- `<script setup lang="ts">` used (no Options API)
- Props/emits are type-based
- Related refs grouped into object refs by context
- Component methods are named functions; inline handlers use arrow functions
- Promise chains (`.then().catch()`) used for Wails binding calls
- Types inline unless shared across files (then in `types/`)
- Flex/grid + Tailwind gap utilities for layout

**Backend (Go):**
- No CGO introduced
- Schema changes in migration files, not raw `AutoMigrate`
- New Wails bindings added to `Bind`/`EnumBind`
